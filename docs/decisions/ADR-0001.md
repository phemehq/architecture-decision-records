## Context and Problem Statement

Pheme's Metric Bridge is the ingest path for all Prometheus `remote_write`
traffic. It must receive, parse, and route every metric sample flowing through
the pipeline in real time. Prometheus clusters at moderate scale write between
50,000 and 500,000 samples per second; at peak, individual series may fan out
across thousands of time-series models updating concurrently.

The Bridge must be:

- **High-throughput** — handle ≥100k samples/sec on a single node without
  becoming the bottleneck in the observability stack.
- **Low-latency** — process each sample within the same scrape interval window
  (typically 15–60 seconds) so anomaly events are near-real-time.
- **Memory-efficient** — hold an in-memory rolling buffer of recent samples per
  series; at 10,000 active series this budget is in the hundreds of MiB.
- **Operationally simple** — ship as a single binary; no runtime dependency
  management; easy to containerise and deploy with a Helm chart.

What language and runtime should the Metric Bridge be written in?

## Decision Drivers

- Throughput and low per-request overhead: sample processing must not gate on
  garbage collection pauses at the p99.
- Native Prometheus ecosystem: the official `client_golang` and
  `prometheus/prometheus` libraries give direct access to the remote_write
  protocol implementation without re-implementing a wire format.
- Concurrency model: per-series ETS model updates are embarrassingly parallel;
  the runtime must make cheap, high-fan-out concurrency ergonomic.
- Single binary deployment: simplifies container images, Helm charts, and
  debugging in production.
- Team capability: the founding team has production Go and Python experience;
  neither Rust nor JVM languages are in the existing skill set.

## Considered Options

- **Go**
- Python (asyncio / multiprocessing)
- Rust
- Java / JVM (Kotlin or Java)

## Decision Outcome

Chosen option: **Go**, because it is the only option that simultaneously
satisfies the throughput requirement, the native Prometheus ecosystem
requirement, and the operational simplicity requirement — while remaining within
the team's existing capability.

### Consequences

- **Good** — `prometheus/prometheus` remote_write server code and
  `client_golang` are used directly; no custom wire-format parsing.
- **Good** — goroutines make it natural to maintain one lightweight coroutine
  per active series for model updates; no thread pool management.
- **Good** — compiles to a static binary; the Docker image is a single
  `FROM scratch` layer.
- **Good** — Go's escape analysis and low-pause GC (sub-millisecond with tuning)
  keep p99 latency predictable at high sample rates.
- **Bad** — the AI layer (Incident Summarizer, Runbook RAG) must run in Python,
  as the ML and LLM SDK ecosystem is Python-native. This necessitates a
  **two-process architecture** (Go bridge + Python sidecar) communicating over
  gRPC, adding operational complexity relative to a single-language stack.
- **Bad** — contributors unfamiliar with Go will face a steeper on-ramp than
  they would with Python.

### Confirmation

The decision holds as long as:

- The Bridge sustains ≥100k samples/sec at ≤50ms p99 ingest latency on a
  2-vCPU/2GiB node (verified by the load test in `tests/loadtest/`).
- The Docker image remains a single binary without a managed runtime.

Re-evaluate if: Rust toolchain maturity in the Prometheus ecosystem catches up
to Go (tracked in the CNCF landscape), or if the two-process gRPC boundary
becomes a significant source of operational incidents.

## Pros and Cons of the Options

### Go

The dominant language in the Prometheus/Kubernetes/CNCF ecosystem. The
`prometheus/prometheus` repository, `client_golang`, and every major
exporter are written in Go.

- **Good** — the remote_write 2.0 protocol parsing is already implemented in
  `prometheus/prometheus` and can be imported directly.
- **Good** — goroutines are cheap (≈2 KiB stack vs ≈2 MiB OS thread); 10,000
  concurrent per-series goroutines are routine.
- **Good** — static binaries eliminate `LD_LIBRARY_PATH` and runtime
  version drift in containers.
- **Good** — GC pauses are bounded and tunable; the runtime is designed for
  server workloads.
- **Neutral** — requires the AI layer to live in a separate Python process.
- **Bad** — the type system, while safe, is less expressive than Rust or
  Haskell for certain data pipeline patterns.

### Python (asyncio / multiprocessing)

Python is used for the AI layer because of the ML/LLM SDK ecosystem. Extending
it to the Bridge would create a single-language stack.

- **Good** — the same language spans Bridge and AI layer, reducing context
  switching for contributors.
- **Good** — rapid prototyping; less boilerplate for data transformation.
- **Bad** — the GIL prevents true CPU parallelism for per-series model
  updates; multiprocessing would be needed, adding IPC complexity.
- **Bad** — asyncio throughput tops out well below the 100k samples/sec
  target without significant sharding and process management.
- **Bad** — no native remote_write 2.0 implementation; would require
  maintaining a custom protobuf parser.
- **Bad** — container images require a Python runtime (tens of MiB overhead
  vs a static Go binary).

### Rust

Rust offers the highest raw throughput and the strongest memory safety
guarantees of any systems language.

- **Good** — zero-cost abstractions and no GC give the lowest possible p99
  latency.
- **Good** — memory safety without a garbage collector eliminates an entire
  class of production incidents.
- **Neutral** — the Prometheus ecosystem in Rust exists (`prometheus` crate)
  but is less mature than `client_golang`.
- **Bad** — no founding team member has production Rust experience; the
  learning curve would significantly delay the MVP.
- **Bad** — Rust compile times and borrow-checker friction slow iteration
  speed during early-stage development.

### Java / JVM (Kotlin or Java)

JVM languages have mature concurrency tooling (virtual threads in Java 21,
coroutines in Kotlin) and a large ecosystem.

- **Good** — Project Loom virtual threads (Java 21+) make per-series
  concurrency straightforward.
- **Good** — a large pool of experienced JVM engineers exists in the broader
  market.
- **Bad** — JVM startup time and warm-up behaviour complicate container
  environments and Kubernetes pod scheduling.
- **Bad** — JVM heap tuning adds operational complexity; GC pauses at high
  allocation rates require careful configuration.
- **Bad** — no native Prometheus remote_write library at the same maturity as
  `client_golang`; significant integration work required.
- **Bad** — no JVM experience in the founding team.
