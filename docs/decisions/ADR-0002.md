# ADR-0002: Use Holt-Winters ETS for Baseline Anomaly Detection

## Context and Problem Statement

Pheme must determine, for each metric series arriving over `remote_write`,
whether a given sample is anomalous. An anomalous sample is one that deviates
meaningfully from what the series would be expected to show at that point in
time given its history.

The detector must:

- Run **per series** — at 10,000 active series, 10,000 model instances execute
  concurrently. Per-model compute and memory cost must be very low.
- Learn **online** — models must update incrementally with each new sample.
  Periodic batch retraining is not acceptable because it introduces latency
  between a metric's behaviour changing and the model adapting.
- Handle **seasonality** — most infrastructure metrics (CPU, request rate,
  latency) exhibit daily and weekly periodic patterns. A detector that ignores
  seasonality will fire false positives every morning when traffic ramps up.
- Be **interpretable** — when Pheme fires an anomaly event, an on-call engineer
  must be able to understand *why*. A model that produces a score without
  explainable bounds is not acceptable in an operational context.
- **Cold-start quickly** — after a deployment or restart, models must reach
  useful accuracy within a few hours of history, not days.
- Produce **calibrated confidence intervals** — the output must be a band
  (expected ± interval) so downstream logic can reason about *how anomalous*
  a sample is, not just *whether* it is anomalous.

What algorithm should Pheme use for per-series anomaly detection?

## Decision Drivers

- Per-instance resource cost must allow 10,000 concurrent models on a 2-vCPU
  node (rules out any neural architecture).
- Online learning is mandatory (rules out algorithms requiring batch retraining).
- Seasonal decomposition is required (rules out pure-ARIMA and Z-score).
- Operator interpretability is a hard requirement (rules out black-box ML
  approaches as the primary detector).
- The founding team has statistics and classical time-series knowledge; deep
  learning expertise is not available in the short term.

## Considered Options

- **Holt-Winters Exponential Triple Smoothing (ETS)**
- Facebook Prophet
- ARIMA / SARIMA
- Z-score / rolling standard deviation
- Isolation Forest
- LSTM / Transformer neural anomaly detector

## Decision Outcome

Chosen option: **Holt-Winters ETS**, because it is the only evaluated algorithm
that simultaneously satisfies online learning, seasonal decomposition,
interpretable confidence intervals, and the per-instance resource budget.

### Consequences

- **Good** — each model is three floating-point scalars (level, trend,
  seasonality component) plus a seasonal buffer. Memory per model is on the
  order of kilobytes; 10,000 models fit comfortably in tens of MiB.
- **Good** — each model update is O(1): a small number of multiply-add
  operations. No matrix operations, no GPU, no batch pipeline.
- **Good** — the confidence interval is directly computed from the model's
  smoothed residuals; operators can read the bounds and understand what
  "expected" means.
- **Good** — the model reaches useful accuracy after one full seasonal cycle
  (24 hours for daily seasonality) of live data, or immediately when
  bootstrapped from Thanos history.
- **Bad** — Holt-Winters assumes **additive** (or multiplicative, with a
  configuration choice) seasonality. It does not model structural breaks —
  if a service is permanently re-provisioned and its baseline shifts, the model
  will take several cycles to adapt and may produce false positives in the interim.
- **Bad** — it does not model non-linear interactions between series. A
  composite anomaly that is only visible across multiple metrics simultaneously
  cannot be detected by per-series ETS alone (a future ADR will address
  multivariate detection).
- **Bad** — three smoothing parameters (alpha, beta, gamma) must be tuned per
  series or set globally. Poorly chosen parameters either miss anomalies or
  produce noise. The initial implementation will use a fixed default; adaptive
  per-series parameter fitting is deferred.

### Confirmation

The decision holds as long as:

- False positive rate (anomaly events that resolve without a corresponding
  human-acknowledged incident) remains below 20% over a rolling 7-day window,
  measured in production via the `pheme_anomaly_false_positive_rate` metric.
- Median detection lead time (time between anomaly event and the first Prometheus
  threshold alert for the same incident) is positive — i.e., Pheme fires first
  in the majority of incidents.

Re-evaluate if: the false positive rate consistently exceeds 20%, or if the
per-series model fan-out approaches the resource budget ceiling. At that point
Prophet (batched, more accurate) or a lightweight learned threshold approach
should be benchmarked against ETS on real production data.

## Pros and Cons of the Options

### Holt-Winters Exponential Triple Smoothing (ETS)

A classical statistical forecasting algorithm that decomposes a series into
level, trend, and seasonal components and updates all three with each new
observation using configurable smoothing weights.

- **Good** — direct online algorithm; no batch step.
- **Good** — explicitly models seasonality at one or more periods.
- **Good** — the prediction interval is a first-class output.
- **Good** — model state is three scalars; memory is negligible.
- **Good** — well-understood and extensively documented in the operations and
  statistics literature.
- **Neutral** — parameter tuning (alpha, beta, gamma) requires either domain
  knowledge or automated fitting; wrong parameters degrade accuracy significantly.
- **Bad** — cannot model sudden structural changes without a reset mechanism.
- **Bad** — univariate model; cross-series anomalies are invisible to it.

### Facebook Prophet

A Bayesian structural time-series model designed for business forecasting with
strong seasonal handling, holiday effects, and automatic change-point detection.

- **Good** — change-point detection handles structural breaks automatically.
- **Good** — models multiple seasonalities (daily, weekly, yearly) without
  explicit period configuration.
- **Bad** — batch algorithm; requires a full refit of the Stan MCMC model to
  incorporate new data. A 15-second scrape interval would require continuous
  retraining, which is computationally prohibitive at 10,000 series.
- **Bad** — each Prophet model is heavy: tens of MiB of Stan state per instance.
  10,000 models would require hundreds of GiB.
- **Bad** — Python-only; the Bridge is written in Go (see ADR-0001), requiring
  an IPC round-trip per sample per series.

### ARIMA / SARIMA

Autoregressive Integrated Moving Average with optional Seasonal component.

- **Good** — SARIMA explicitly models seasonal structure.
- **Good** — well-studied and widely understood.
- **Bad** — batch model; does not support online (incremental) updates.
- **Bad** — model order selection (p, d, q, P, D, Q, s) requires automated
  search (`auto_arima`), which is expensive at 10,000 series.
- **Bad** — Python-only, same IPC penalty as Prophet.

### Z-score / Rolling Standard Deviation

Treat the rolling mean and standard deviation of a configurable window as the
expected range. Flag samples more than N standard deviations from the mean.

- **Good** — extremely simple to implement and explain.
- **Good** — O(1) update, negligible memory.
- **Bad** — does not model seasonality. A metric that spikes every morning will
  constantly trigger false positives at the start of each day.
- **Bad** — assumes a normal distribution, which most infrastructure metrics do
  not follow (heavy right tail for latency/error rate metrics).
- **Bad** — window length is a global parameter; different series have different
  natural time scales.

### Isolation Forest

An ensemble of random decision trees that isolates anomalies by randomly
partitioning the feature space.

- **Good** — handles multivariate inputs naturally.
- **Good** — does not assume a distribution.
- **Bad** — requires a training corpus and periodic batch retraining.
- **Bad** — does not model seasonality or trends.
- **Bad** — anomaly score is not a calibrated probability or interval;
  threshold selection for the score is arbitrary.
- **Bad** — not interpretable in operational terms — cannot tell an operator
  *what* was expected vs *what* was observed.

### LSTM / Transformer Neural Anomaly Detector

Sequence-to-sequence neural models trained to reconstruct expected metric
sequences; anomalies are detected as high reconstruction error.

- **Good** — can learn complex non-linear patterns and multi-series correlations.
- **Good** — generalises across metric types without per-series parameter tuning.
- **Bad** — training requires a GPU and a significant history corpus.
- **Bad** — inference is not O(1); even a small LSTM adds significant CPU cost
  per sample at 10,000-series fan-out.
- **Bad** — reconstruction error is not a calibrated confidence interval.
- **Bad** — black box from an operator's perspective.
- **Bad** — no founding team member has the deep learning expertise to build
  and maintain a production neural anomaly detector at this stage.
